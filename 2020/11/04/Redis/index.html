Redis

不同于传统数据库，redis的数据存在内存中，因此读写速度非常快。因此能支持高性能（直接操作内存）和高并发。

支持的数据类型

  • String:
    • set、get等
  • Hash:
    • hget、hset、hgetall等
  • List:
    • lpush、rpush、lpop、rpop、lrange等
    • redis list的实现为一个双向链表
  • Set：
    • sadd、spop、smembers、sunion等
    • 基于set可以轻易实现交集、并集、差集操作。
  • Sorted Set：
    • zadd、zrange、zrem、zcard等
    • 和set相比，sorted set增加了一个权重参数score，使得集合中的元素能够按照score进行有序排列。

本地缓存&分布式缓存

本地缓存

如java的map或guava实现的本地缓存。

  • 轻量、快速，生命周期随着jvm的销毁而结束。
  • 多实例的情况下，每个实例都需要各自保存一份缓存，缓存不具有一致性。

分布式缓存

  • 多实例情况下，共用一份缓存，缓存具有一致性。
  • 需要保证服务的高可用，架构更复杂。
  • 可以持久化
  • 有缓存过期机制
  • 有丰富的API，支持更多数据结构。

Redis线程模式

文件事件处理器（File Event Handler)

Redis基于Reactor模式开发了网络事件处理器，这个处理器叫文件事件处理器，是单线程的。采用IO多路复用机制同时监听多个socket，根据socket上的事件来选择对应的事件处理器。

文件事件处理器的结构包含4个部分：

  • 多个socket
  • I/O多路复用程序
  • 文件事件分派器
    接收I/O多路复用传来的套接字，并根据套接字产生的事件的类型，调用相应的时间处理器。
  • 事件处理器（连接应答处理器、命令请求处理器、命令回复处理器）

基于事件驱动，适合处理海量的I/O事件

I/O多路复用

I/O就是指的我们网络I/O，多路指多个TCP连接（或多个Channel），复用指复用一个或少量线程。串起来理解就是，很多个网络I/O复用一个或少量的线程来处理这些连接。
解决单个服务器应对多客户端同时IO请求阻塞问题的方案

  • 具体做法：调用一个阻塞方法，这个阻塞方法会监听跟踪每一个IO流的状态，当有一个新的数据传输请求到来，就会通知服务器，然后服务器找到对应有需求的客户端，并读取它要传输的数据。
  • 单线程或单进程同时监控若干个文件描述符是否可以执行IO操作能力。
  • 阻塞VS和非阻塞：调用一个IO操作能不能立即得到返回应答，如果不能立即获得返回，需要等待，那就阻塞了；否则就可以理解为非阻塞。描述的是用户线程调用内核IO操作的方式。阻塞是指IO操作需要彻底完成后才返回到用户空间，而非阻塞是指IO操作被调用后立即返回给用户一个状态值，无需等到IO操作彻底完成。
  • 同步VS异步：描述的是用户线程与内核的交互方式。同步是指用户线程发起IO请求后需要等待或者轮训内核IO操作完成后才能继续执行；而异步是指用户线程发起IO请求后仍继续执行，当内核IO操作完成后会通知用户线程，或者调用用户线程注册的回调函数。

为什么Redis使用单线程模型还能保证高性能

  1. 纯内存操作
  2. 基于非阻塞IO多路复用
  3. 单线程避免线程切换

Redis和Memcacahed的区别

  1. redis支持更丰富的数据类型（支持更复杂的应用场景）；
  2. redis支持数据持久化；
  3. redis原生支持集群模式；
  4. memcached是多线程、非阻塞IO复用复用的网络模型，redis使用单线程的IO多路复用。

Redis过期策略和内存淘汰机制

过期策略

  • 定期删除：
    • redis默认每隔100ms就随机抽取一些设置了过期时间的key，检查是否过期，如果过期就删除。
    • 会导致很多过期key到了时间并没有被删除掉
  • 惰性删除：
    • 在获取某个key的时候，redis会检查一下，该key是否设置了过期时间，如果设置了过期时间那么是否过期了。过期则删除，不会返回任何东西。

通过上述两种手段结合起来，实际上还是有问题的。如果定期删除漏掉了很多过期的key，然后我们也没有主动及时去查，没走惰性删除，会导致大量过期key堆积在内存里，导致redis内存块耗尽。

内存淘汰机制

  1. volatile-lru：从已设置过期时间的数据集中挑选最近最少使用的数据淘汰。
  2. volatile-ttl：从已设置过期时间的数据中挑选将要过期的数据淘汰。
  3. volatile-random：从已设置过期时间的数据集中任意选择数据淘汰。
  4. allkeys-lru：当内存不足以容纳写入数据时，在键空间中，移除最近最少使用的key（最常用）。
  5. allkeys-random：从数据集中任意选择数据淘汰。
  6. no-eviction：禁止驱逐数据，当内存不足以容纳新写入数据时，新写入操作会报错。
  7. volatile-lfu：从已过期时间的数据集中挑选最不经常使用的数据淘汰。
  8. allkeys-lfu：当内存不足以容纳新写入数据时，在键空间中，移除最不经常使用的key。

Redis持久化

RDB

快照持久化（Redis DataBase），通过创建快照来获得存储在内存里面的数据在某个时间点上的副本，即在不同的时间点，将Redis内存的数据生成快照并存储到磁盘等介质上。是Redis默认采用的持久化方式。

AOF

将Redis执行过的所有写指令记录下来，在Redis重启时，只要把这些写指令从前到后再执行一遍，就可以实现数据恢复。AOF(Append-only file)持久化的实时性更好，默认情况下没有开启，可以通过appendonly参数开启：

开启AOF持久化后每执行一条会更改Redis中的数据的命令，Redis就会将该命令写入硬盘中的AOF文件。默认文件名是appendonly.aof，默认AOF持久化策略是每秒中执行一次fsync（fsync是指把缓存中的写指令记录到磁盘中）。

  • AOF文件重写机制：即当AOF文件的大小超过所设定的阈值时，redis就会启动AOF文件的内容压缩，只保留可以恢复数据的最小指令集。举个例子或许更形象，假如我们调用了100次INCR指令，在AOF文件中就要存储100条指令，但这明显是很低效的，完全可以把这100条指令合并成一条SET指令，这就是重写机制的原理。
  • 对比：
    • AOF文件要比RDB文件大
    • AOF恢复速度不如RDB

缓存穿透

大量请求的key不存在与缓存中，导致请求到了数据库上。

解决方法

布隆过滤器

它本身是一个很长的二进制向量（想象成数组）和一系列随机映射函数（想象成多个hash函数），二进制向量中存放的不是0就是1.

  • 布隆过滤器可以确定一个元素肯定不存在，但是不能确定一个元素肯定存在，因此存在误判率。
  • 可以添加元素，但是不能删除元素（除非添加计数器）。因为不同值的hash值可能相同。
  • 使用布隆过滤器过滤掉肯定不存在的数据，误判的请求虽然还是会放到数据库，但已经极大减少穿透的数量。

缓存无效的key

如果缓存和数据库都查不到某个key的数据就写一个到redis中去冰设置过期时间。尽量将无效key的过期时间设置短一点，避免redis中缓存大量无效的key。

缓存击穿

是指一个key非常热点，在不停的扛着大并发，大并发集中对这一个点进行访问，当这个key在失效的瞬间，持续的大并发就穿破缓存，直接请求数据库，好像蛮力击穿一样。

解决方法

  • 延长热点key的过期时间；
  • 利用互斥锁保证同一时刻只有一个客户端可以查询底层数据库的这个数据，一旦查到数据就缓存至Redis内，避免其他大量请求同时穿过Redis访问底层数据库；

缓存雪崩

缓存同⼀时间⼤⾯积的失效，所以，后⾯的请求都会落到数据库上，造成数据库短时间内承受⼤ 量请求⽽崩掉。

解决方法

  • 延长热点key的过期时间；
  • 在可接受的时间范围内，随机设置key的过期时间

多线程时代

Redis6版本引入了多线程。单线程的性能瓶颈主要在网络IO操作上，也就是在读写网络read/write系统调用执行期间会占用大部分CPU时间。如果要对大的键值对进行删除操作的话，在短时间内是删不完的，那么对于单线程来说就会阻塞后边的操作。

Redis 在设计上采用将网络数据读写和协议解析通过多线程的方式来处理，对于命令执行来说，仍然使用单线程操作。
